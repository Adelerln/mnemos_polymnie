Pourquoi utiliser Git ?

Ce que Git apporte à l'équipe

Ce que Git permet :

  • Branches isolées — chacun travaille sans gêner les autres
  • Relecture obligatoire — tout code est vérifié avant fusion
  • Historique complet — qui a changé quoi, quand, pourquoi
  • Retour en arrière — en une commande si quelque chose casse
  • Détection de conflits — Git prévient quand deux personnes modifient le même fichier
💡 Git = le système de versioning (local, sur ton ordi).
💡 GitHub = la plateforme en ligne qui héberge le dépôt et ajoute PR, issues, CI…

Créer une branche & faire une PR

Tout se fait depuis GitHub Desktop + le site GitHub

🌿 Dans GitHub Desktop

  1. Fetch / Pull — cliquer sur "Fetch origin" puis "Pull" pour se mettre à jour
  2. Nouvelle branche — Current Branch → New Branch → nommer familles/ajout-fiche-sanitaire
  3. Coder — travailler dans VS Code normalement
  4. Commit — dans GitHub Desktop, cocher les fichiers → écrire le message → "Commit to…"
  5. Push — cliquer sur "Push origin"

🔀 Sur GitHub (site web)

  1. Un bandeau jaune apparaît : "Compare & pull request"
  2. Vérifier : base = develop, compare = ta branche
  3. Titre clair : « Familles — Ajout fiche sanitaire »
  4. Décrire ce que tu as fait en quelques lignes
  5. Assigner un relecteur
  6. Cliquer "Create pull request"
Après relecture approuvée → Merge pull request → Supprimer ta branche.

Nos règles de code

Le code est lu beaucoup plus souvent qu'il n'est écrit

🎯 Les 5 règles à retenir

  • ~200 lignes max par fichier — au-delà, découper
  • 1 fichier de types par domaine (types/family.ts, types/centre.ts…)
  • 1 seule méthode CSS — Tailwind uniquement
  • Commenter le pourquoi, pas le quoi — code lisible avant tout
  • 0 doublon — chaque logique n'existe qu'à un seul endroit
💡 Écrire un code propre, c'est un cadeau pour ton futur toi et pour l'équipe.

DRY — Don't Repeat Yourself

Si tu copies-colles → crée une fonction réutilisable

✅ La bonne approche

// lib/phone.ts — la logique n'existe qu'ici
export function formatPhone(value: string) {
  return value
    .replace(/\D/g, "")
    .replace(/(\d{2})(?=\d)/g, "$1 ");
}

// Dans n'importe quelle page :
import { formatPhone } from "@/lib/phone";
const phone = formatPhone(value);

→ Un bug ? On corrige 1 seul fichier, tout le projet est réparé.

💡 Le réflexe : même logique utilisée 2 fois ? → extraire dans lib/ ou un composant dans components/.

L'IA n'est pas magique

Un outil puissant, mais qui a ses limites

✅ L'IA est bonne pour…

  • Générer du code répétitif rapidement
  • Expliquer un concept ou un bout de code
  • Proposer une structure de fichiers/dossiers
  • Débugger un message d'erreur précis
  • Convertir entre formats (camelCase ↔ snake_case…)

⚠️ L'IA est mauvaise pour…

  • Comprendre le contexte métier du projet sans explication
  • Garantir que le code fonctionne sans le tester
  • Prendre des décisions d'architecture à ta place
  • Écrire du code sécurisé sans vérification
  • Remplacer la réflexion et la planification
💡 Règle d'or : L'IA est un assistant, pas un développeur. C'est toi qui décides, vérifies et valides.

Planifier avant de prompter

Générer un fichier plan, puis coder étape par étape

La méthode : fichier plan → implémentation

  1. Comprendre ce qu'on veut — résultat attendu, fichiers existants concernés
  2. Demander à l'IA de générer un fichier plan (ex : PLAN_inscriptions.md) contenant le contexte complet : objectif, fichiers à créer/modifier, types, dépendances
  3. Implémenter étape par étape — un prompt = une sous-tâche du plan, valider avant de continuer
💡 Pourquoi un fichier plan ? L'IA peut le relire à chaque prompt pour garder le contexte.
→ "Consulte PLAN_inscriptions.md et implémente l'étape 3."
⚠️ À chaque nouvelle implémentation, rappelle à l'IA :
  • Ne pas recoder ce qui existe déjà
  • Commenter le code (expliquer le pourquoi)
  • Prévoir la gestion d'erreurs (try/catch, messages utilisateur)
  • Prévoir les tests

Écrire de bons prompts

Clair, simple, avec du contexte

Les 4 ingrédients d'un bon prompt

1. Le contexteDans quel fichier, quel framework, quelle techno
2. La tâcheCe que tu veux, en une phrase
3. Les contraintesFormat attendu, conventions, limites
4. Un exempleUn résultat attendu ou un modèle à suivre

Exemple concret :

"Dans le fichier services/families.ts (Next.js + Supabase),
crée une fonction fetchFamilyById(id: number) qui :
- récupère la famille avec ses adultes et enfants
- retourne un objet typé FamilyWithRelations
- gère les erreurs avec un try/catch
Utilise le même style que fetchAllFamilies déjà dans le fichier."
💡 Astuce : Plus ton prompt est précis, moins tu auras de corrections à faire après.

Toujours vérifier

Ne jamais copier-coller aveuglément

Checklist avant d'intégrer du code IA

  1. Lire et comprendre — si tu ne comprends pas le code, ne l'utilise pas
  2. Tester — est-ce que ça fonctionne vraiment ? lancer le projet, vérifier dans le navigateur
  3. Vérifier la sécurité — pas de clé en dur ? pas de faille ? pas de any partout ?
  4. Vérifier le style — respecte-t-il nos conventions ? (nommage, structure, Tailwind…)
  5. Adapter — le code IA est un point de départ, pas un résultat final
⚠️ Pièges courants du code IA :
  • Invente des fonctions ou des API qui n'existent pas
  • Oublie la gestion d'erreurs
  • Utilise des packages non installés
  • Ignore les conventions du projet
